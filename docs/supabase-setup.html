0. 目标与架构

本系统把 GUCC 的数据库管理前端作为静态页面放在 GitHub Pages，通过 Supabase Auth 登录，调用 Supabase Edge Function，再由 Edge Function 调用 Postgres RPC 读写多张业务表。

GitHub Pages 静态前端
  apps/command-center/index.html
  apps/command-center/src/*.js
  apps/command-center/styles/app.css
        ↓
Supabase Auth
  登录、获取 access_token
        ↓
Supabase Edge Function
  /functions/v1/gameup-api
  校验 access_token
  校验 public.app_users 白名单
  调用 PostgreSQL RPC
        ↓
Supabase Postgres
  games / characters / character_progress / parties
  game_versions / resources / resource_relations 等表

原则：GitHub 只放前端和公开配置；真正的写入逻辑放在 Postgres RPC；鉴权边界放在 Edge Function；service_role key 不能进入 GitHub、前端、README 或截图。

3. Supabase SQL：检查结构、安装 RPC 和约束

3.1 结构检查

打开：

Supabase Dashboard
→ Project
→ SQL Editor
→ New query

运行结构检查 SQL：

sql/01_check_schema.sql 或 supabase/sql/01-check-schema.sql

结果含义处理
OK核心表和列基本符合预期继续安装后端 RPC
MISSING_COLUMNS列名与 RPC 预期不一致调整 SQL 或修改 RPC 函数中的列名
MISSING_TABLE核心表缺失先补导入对应模块

3.2 安装后端 RPC + app_users 白名单表

继续运行后端安装 SQL：

sql/02_app_backend_jsonb_rpc.sql 或 supabase/sql/02-install-command-center.sql

它应创建或更新：

public.app_users register_app_owner(email) app_search_characters(jsonb) app_save_character(jsonb) app_delete_character(jsonb) app_search_parties(jsonb) app_save_party(jsonb) app_search_versions(jsonb) app_search_resources(jsonb)

注意：如果安装时出现重复 URL、重复 relation 或 ON CONFLICT 相关错误，先执行本文附录中的“数据库约束修复 SQL”，再重新运行本步骤。

3.3 SQL 级别快速测试

select public.app_search_characters('{"keyword":"千","limit":3}'::jsonb);

如果返回 JSON 数组，说明 RPC 至少可以被数据库直接调用。

5. Supabase Edge Function 部署

5.1 安装并使用 Supabase CLI

推荐使用项目本地 CLI，不依赖全局 supabase 命令。

cd "C:\Users\miket\Git管理下\GUCC"

node -v
npm -v

npm init -y
npm install supabase --save-dev

npx supabase --version
npx supabase login
npx supabase link --project-ref rubjeqnuxuvupjwyksmo

如果直接运行 supabase login 报 The term 'supabase' is not recognized，说明没有全局命令。使用 npx supabase login 即可。

5.2 设置 Edge Function Secrets

当前 Supabase CLI 不允许手动设置以 SUPABASE_ 开头的保留变量。托管 Edge Function 默认可访问 SUPABASE_URL 和 SUPABASE_SERVICE_ROLE_KEY 等环境变量。

因此只设置自定义变量：

npx supabase secrets set ALLOWED_ORIGINS=http://localhost:8000,https://inertia77.github.io

不要执行：npx supabase secrets set SUPABASE_SERVICE_ROLE_KEY=...。不要把 service role key 写入前端、GitHub 或文档。

5.3 部署 gameup-api

npx supabase functions deploy gameup-api --no-verify-jwt
npx supabase functions list

部署成功后应看到：

NAME       SLUG       STATUS
gameup-api gameup-api ACTIVE

函数地址格式：

https://rubjeqnuxuvupjwyksmo.supabase.co/functions/v1/gameup-api

12.1 权限修复 SQL

当 Edge Function 可以被调用，但返回 permission denied for table app_users 或业务表权限错误时使用。

begin;

grant usage on schema public to service_role;

grant select, insert, update, delete on table public.app_users to service_role;

grant select, insert, update, delete on table public.games to service_role;
grant select, insert, update, delete on table public.characters to service_role;
grant select, insert, update, delete on table public.character_names to service_role;
grant select, insert, update, delete on table public.character_progress to service_role;
grant select, insert, update, delete on table public.character_evaluations to service_role;
grant select, insert, update, delete on table public.parties to service_role;
grant select, insert, update, delete on table public.party_members to service_role;
grant select, insert, update, delete on table public.game_versions to service_role;
grant select, insert, update, delete on table public.version_banners to service_role;
grant select, insert, update, delete on table public.resources to service_role;
grant select, insert, update, delete on table public.resource_relations to service_role;

grant usage, select on all sequences in schema public to service_role;
grant execute on all functions in schema public to service_role;

notify pgrst, 'reload schema';

commit;

12.2 数据库约束修复 SQL

当安装 RPC 或保存角色时遇到重复 URL、重复 relation、或 ON CONFLICT 约束不匹配时使用。

执行前建议先确认当前数据已备份。该 SQL 会删除重复的资源和重复关系，只保留规范化后的唯一记录。

begin;

-- 1. Normalize URL values.
update public.resources
set url = null
where url is not null
  and trim(url) = '';

update public.resources
set url = trim(url)
where url is not null
  and url <> trim(url);

-- 2. Remove older partial index if it cannot match ON CONFLICT (url).
drop index if exists public.ux_resources_url;

-- 3. Build a map from duplicated resources to the resource row to keep.
create temp table _resource_dedupe_map as
with ranked as (
  select
    id,
    url,
    first_value(id) over (
      partition by url
      order by created_at nulls last, id
    ) as keep_id,
    row_number() over (
      partition by url
      order by created_at nulls last, id
    ) as rn
  from public.resources
  where url is not null
)
select
  id as old_id,
  keep_id as new_id,
  url
from ranked
where rn > 1;

-- 4. Copy old relations onto the kept resource only when the same relation does not already exist.
insert into public.resource_relations (
  resource_id,
  entity_type,
  entity_id,
  relation_type,
  source_sheet,
  source_field
)
select
  m.new_id,
  rr.entity_type,
  rr.entity_id,
  rr.relation_type,
  rr.source_sheet,
  rr.source_field
from public.resource_relations rr
join _resource_dedupe_map m
  on rr.resource_id = m.old_id
where not exists (
  select 1
  from public.resource_relations x
  where x.resource_id = m.new_id
    and x.entity_type is not distinct from rr.entity_type
    and x.entity_id is not distinct from rr.entity_id
    and x.relation_type is not distinct from rr.relation_type
    and x.source_sheet is not distinct from rr.source_sheet
    and x.source_field is not distinct from rr.source_field
);

-- 5. Delete relations and resource rows pointing to duplicated resources.
delete from public.resource_relations rr
using _resource_dedupe_map m
where rr.resource_id = m.old_id;

delete from public.resources r
using _resource_dedupe_map m
where r.id = m.old_id;

drop table _resource_dedupe_map;

-- 6. Remove fully duplicated relation rows.
delete from public.resource_relations rr
using (
  select
    ctid,
    row_number() over (
      partition by
        resource_id,
        entity_type,
        entity_id,
        relation_type,
        source_sheet,
        source_field
      order by id
    ) as rn
  from public.resource_relations
) d
where rr.ctid = d.ctid
  and d.rn > 1;

-- 7. Remove basic duplicated relation rows.
delete from public.resource_relations rr
using (
  select
    ctid,
    row_number() over (
      partition by
        resource_id,
        entity_type,
        entity_id,
        relation_type
      order by id
    ) as rn
  from public.resource_relations
) d
where rr.ctid = d.ctid
  and d.rn > 1;

-- 8. Create unique indexes that match the RPC ON CONFLICT clauses.
create unique index ux_resources_url
on public.resources (url);

create unique index if not exists ux_resource_relations_full
on public.resource_relations (
  resource_id,
  entity_type,
  entity_id,
  relation_type,
  source_sheet,
  source_field
);

create unique index if not exists ux_resource_relations_basic
on public.resource_relations (
  resource_id,
  entity_type,
  entity_id,
  relation_type
);

notify pgrst, 'reload schema';

commit;

12.3 检查索引

select
  schemaname,
  tablename,
  indexname,
  indexdef
from pg_indexes
where schemaname = 'public'
  and (
    indexname = 'ux_resources_url'
    or indexname = 'ux_resource_relations_full'
    or indexname = 'ux_resource_relations_basic'
  )
order by tablename, indexname;

结束标准

当 GitHub Pages 线上页面可以打开，用户可以登录，测试后端连接返回 pong: true，角色搜索和保存可以正常工作，并且 Git 仓库没有敏感 key 与本地临时目录时，部署完成。

这套结构后续扩展时仍保持同一原则：GitHub Pages 负责界面，Supabase Auth 负责登录，Edge Function 负责安全边界，Postgres RPC 负责多表业务写入。🏯
